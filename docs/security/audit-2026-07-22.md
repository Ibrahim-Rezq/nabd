# Security audit — nabd

**Date:** 2026-07-22
**Scope:** whole application (web PWA + Capacitor Android shell + Supabase backend integration).
**Method:** full-surface read of every security-relevant file — auth/session, sync, native/Android,
secrets/config, client rendering (XSS sinks), service worker/PWA, outbound network/PII,
dependencies. `pnpm audit` for supply chain. Findings below were each confirmed against the
actual source, not inferred.

> Note on method: a multi-agent fan-out was started to cross-verify findings adversarially; the
> account session limit cut it off after the first domain. All findings here were then verified
> manually by reading the referenced code directly. Nothing is speculative.

## The one thing that matters most

**There is no server tier.** No route handlers, no server actions, no `app/api/`, and
`SUPABASE_SERVICE_ROLE_KEY` is never referenced in code. Every Supabase call runs in the browser
(or the APK WebView) with the **anon key**. That means **Supabase Row-Level Security is the only
thing isolating one user's data from another's** — and those policies live in the Supabase
dashboard, not in this repo, so they cannot be reviewed, tested, or reproduced from code. If RLS
is off or misconfigured on the live project, every user's devotional history is readable and
overwritable by any other authenticated user. Verify this first (see F1).

## Findings (by severity)

### HIGH

**F1 — Per-user data isolation rests entirely on unverifiable Supabase RLS**
`features/sync/db.ts` (`pushOutbox` L80, `pullTable` L99–103)
The client upserts rows with a client-chosen `id` (`onConflict: 'id'`) and a client-set
`user_id`, and reads with a client-side `.eq('user_id', userId)`. None of that is a security
boundary — anyone holding the public anon key can issue arbitrary PostgREST requests with any
`id`/`user_id`. The only real control is a Postgres RLS policy enforcing `auth.uid() = user_id`
on SELECT **and** INSERT/UPSERT for `wird_versions` and `wird_entries`. No SQL migration or
policy definition exists anywhere in the repo.
_Impact:_ if RLS is missing/wrong, cross-user read + overwrite (colliding-`id` upsert) of all
wird data. This is critical-if-off, and it's currently unverifiable from code.
_Fix:_ (1) confirm RLS is ON with `using`/`with check (auth.uid() = user_id)` on both tables in
the live project; (2) commit the policies as `supabase/migrations/*.sql` so the control is
version-controlled and reviewable; (3) consider a DB constraint/trigger binding `id` ownership
to `user_id` to prevent cross-user id-collision upserts.

### MEDIUM

**F2 — Open redirect on the web OAuth callback**
`features/auth/hooks/useAuthCallback.ts:28`
`const next = nextParam && nextParam.startsWith('/') ? nextParam : '/'` — `startsWith('/')`
accepts protocol-relative targets like `//evil.com` and `/\evil.com`, which the browser resolves
to an external origin. `router.replace(next)` then redirects the just-authenticated user
off-site (phishing / consent-screen spoofing).
_Fix:_ reject values that start with `//` or `/\`; accept only a single-slash path, e.g.
`nextParam.startsWith('/') && !nextParam.startsWith('//') && !nextParam.startsWith('/\\')`, or
validate with `new URL(nextParam, origin)` and require the result's origin to equal `origin`.

**F3 — No security headers / CSP on the server-rendered web build**
`next.config.ts` (no `headers()`)
No Content-Security-Policy, X-Frame-Options/`frame-ancestors`, Strict-Transport-Security,
Referrer-Policy, or Permissions-Policy. The web build is server-rendered on Vercel and performs
the OAuth code exchange, so it has no clickjacking protection and no CSP to blunt an injected
script.
_Fix:_ add an async `headers()` for the non-native build: a strict CSP, `X-Frame-Options: DENY`
(or `frame-ancestors 'none'`), HSTS, `Referrer-Policy: strict-origin-when-cross-origin`,
`X-Content-Type-Options: nosniff`, and a minimal `Permissions-Policy`. Note: the CSP must allow
the one inline appearance script (`app/layout.tsx:101`) via a hash or nonce — see F-INFO-b.

**F4 — Vulnerable transitive dependencies**
`package.json` / `pnpm-lock.yaml` (`pnpm audit`: 2 high, 1 moderate)

- `sharp <0.35.0` (via `next`) — inherited libvips CVEs (high). Used by `next/image` on the web
  build (the native export sets `images.unoptimized`).
- `postcss <8.5.10` — XSS via unescaped `</style>` in stringify (moderate); build-time only.
- `fast-uri` (via `@sentry/nextjs` → webpack-plugin) — build-time only.
  All are build/server-side, not shipped in the client runtime, so live exposure is limited — but
  they should be patched.
  _Fix:_ bump `next` to a release pulling `sharp >=0.35.0` and `postcss >=8.5.10`, or add
  `pnpm.overrides` to force patched versions; re-run `pnpm audit`.

### LOW

**F5 — Android `allowBackup="true"`**
`android/app/src/main/AndroidManifest.xml`
The app holds a Supabase session (localStorage) and local Dexie data. With `allowBackup="true"`
and no `dataExtractionRules`, that data is extractable via `adb backup` on a device with USB
debugging.
_Fix:_ set `android:allowBackup="false"` (this is an auth-token-bearing app) or define
`android:dataExtractionRules` / `fullBackupContent` that exclude the WebView storage.

**F6 — Precise GPS coordinates sent to a third party without disclosure**
`lib/impure/reverse-geocode.ts:15`
Exact `latitude`/`longitude` are sent to `api.bigdatacloud.net` to resolve a city name. Not
SSRF (hardcoded HTTPS host, numeric coords), but precise location leaves the device to a third
party with no in-code consent/disclosure — a data-minimization gap for a privacy-sensitive
audience.
_Fix:_ truncate coordinates to ~2 decimals (city-level is enough), disclose the third-party
call in the privacy policy, and consider gating behind explicit consent.

**F7 — Custom-scheme OAuth deep link is interceptable**
`AndroidManifest.xml` (`nabd://auth`, exported activity) + `useNativeAuthListener.ts`
Custom schemes aren't verified like App Links, so a malicious app can register `nabd://` too.
PKCE mitigates the real damage — the intercepted `code` is useless without the verifier held in
the legitimate app — so session theft isn't practical, but it's still a weaker binding than
verified App Links.
_Fix (hardening):_ migrate to Android App Links (`https://` + `android:autoVerify="true"` +
`assetlinks.json`), or accept the residual risk and document the PKCE dependency.

**F8 — FileProvider paths rooted at the whole directory**
`android/app/src/main/res/xml/file_paths.xml`
`<external-path path="."/>` and `<cache-path path="."/>` expose the entire external-files and
cache directories. The provider is `exported="false"`, so exposure requires the app to grant a
URI, but the broad root is unnecessary (Capacitor template default).
_Fix:_ narrow the paths to the specific subdirectory the app actually shares.

### INFO / hygiene

**F-INFO-a — `SUPABASE_SERVICE_ROLE_KEY` declared but unused**
`.env.example:14`. No server code consumes it; the "SERVER ONLY (route handlers)" comment
implies a server tier that doesn't exist. Correctly non-`NEXT_PUBLIC_` and `.env.*` is
gitignored, so nothing sensitive is committed. Latent risk: an RLS-bypassing key with no
consumer invites future misuse. _Fix:_ remove until a real server route needs it.

**F-INFO-b — Inline appearance script needs a CSP allowance**
`app/layout.tsx:101` renders `dangerouslySetInnerHTML={{ __html: APPEARANCE_INIT_SCRIPT }}`.
The script (`lib/impure/appearance.ts:53`) is fully static and app-authored — **not an XSS
sink**. Called out only because it will require a hash/nonce when F3's CSP lands.

**F-INFO-c — Leftover Cordova `<access origin="*"/>`**
`android/app/src/main/res/xml/config.xml`. Inert under Capacitor (no `server`/`allowNavigation`
in `capacitor.config.ts`), but should be removed to avoid confusion.

## What's already done right (verified, not assumed)

- **Session validation is correct.** `lib/db/supabase/middleware.ts` calls `getUser()` (which
  revalidates the token with Supabase) with no logic between client creation and the call — the
  recommended SSR pattern. App code uses `getUser()`, not the spoofable `getSession()`.
- **No committed secrets.** `git ls-files` + JWT/`service_role`/private-key grep are clean;
  `.gitignore` covers `.env.*` with a `!.env.example` exception.
- **XSS surface is minimal.** The only `dangerouslySetInnerHTML` is a static app-authored script
  (F-INFO-b). Adhkar/intentions content is static and app-authored — no user-generated content
  reaching a DOM sink, so no stored-XSS vector.
- **PKCE OAuth** with `detectSessionInUrl:false` on native; the `next` redirect is at least
  path-gated (weakened only by F2).
- **Service worker is conservative.** `skipWaiting:false` gated on explicit user opt-in; the
  `push` handler renders notification **text** (not HTML). No SSRF (geocode host hardcoded,
  coords numeric).
- **Sentry sends minimal PII by default** — `sendDefaultPii` not enabled, no session replay
  configured. (Still worth confirming coordinates/city never get attached as event context.)

## How to verify the fixes

- **F1:** in the Supabase SQL editor, `select relname, relrowsecurity from pg_class where
relname in ('wird_versions','wird_entries');` (both must be `t`), and list policies with
  `select * from pg_policies where tablename in ('wird_versions','wird_entries');`. Then, signed
  in as user A, attempt to `select`/`upsert` a row with user B's `user_id`/`id` and confirm it's
  rejected. Add the passing policies as a checked-in migration.
- **F2:** unit-test `useAuthCallback`'s next-param logic with `//evil.com`, `/\evil.com`,
  `https://evil.com`, `/settings` — only the last should survive.
- **F3:** after adding `headers()`, check response headers on a Vercel preview
  (`curl -I`) and load the app to confirm the CSP doesn't break the inline appearance script.
- **F4:** `pnpm audit --prod` returns clean (or only accepted advisories).
- **F5/F7/F8:** rebuild the APK (`pnpm cap:sync`) and inspect the merged manifest / try
  `adb backup` to confirm data is no longer extractable.
