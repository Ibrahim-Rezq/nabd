# ADR-0005 — CI/CD

- **Status:** accepted
- **Date:** 2026-07-13

## Context

We needed a definition of "safe to merge" and "safe to release" that a building AI can run
mechanically, plus a rule that nothing is pushed before it is green locally.

## Decision

**Six required GitHub Actions checks, one per workflow file:** `lint` (incl. import-boundary
rules), `typecheck` (tsc strict), `test` (unit + component), `e2e` (Playwright), `build`
(production build), `colocated-test-check`. All are required to merge. **Local gate:** the same
checks must pass locally (`pnpm lint && pnpm typecheck && pnpm test && pnpm build`, plus e2e
where relevant) before any push — never push to find out. **Pre-production smoke test** on the
staging URL before `dev → master`; **post-deploy verification** re-runs the project success
check on the production URL. Secrets: `.env.local` gitignored, `.env.example` committed and
documented, real values in the Vercel/Supabase dashboards.

Rejected: an informational-only check tier (blurs the green bar); manual production promotion
without a staging smoke test (skips the last human check).

## Consequences

- Merges are blocked until all six checks pass; releases require them twice (feature→dev,
  dev→master).
- The local gate keeps CI from being used as a discovery mechanism.
- `.env.example` documents every variable, including Supabase, Sentry, and analytics keys.
- Deployment is "done" only after the production success check passes.
- Revisit CI duration if the e2e suite grows enough to slow the merge loop.
